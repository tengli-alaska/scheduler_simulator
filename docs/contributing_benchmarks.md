# Contributing Benchmarks (Lean v1)

This project prioritizes low-friction reproducible benchmarking.

## Ground rules

1. Common workflows must stay simple:
   - add a suite in one JSON file
   - run with one command
   - inspect outputs without framework internals
2. Prefer clear CSV outputs over complex internal abstractions.
3. New abstractions must reduce user friction in the next 2–4 weeks.

## Contribution checklist

For new suite/workload/scheduler contributions:

1. Validate suite spec:
   - `python3 scripts/validate_benchmark_spec.py --suite <suite.json>`
2. Dry-run the suite:
   - `python3 -m benchmark.runner.run_suite --suite <suite.json> --dry-run`
3. Run at least one real case and generate manifests.
4. Rebuild analysis outputs:
   - `python3 scripts/aggregate_results.py`
   - `python3 scripts/export_summary_table.py`
5. If plotting deps exist, regenerate experiment figures.
6. Confirm `analysis/quality_checks.csv` has no unexpected `ERROR` rows.

## Naming conventions

- Suite ids: lowercase kebab-case (`my-suite-v1`)
- Experiment ids: descriptive snake/kebab prefixed by `expN_` when comparable
- Run ids: generated by runner (do not handcraft)

## Extensibility rules

- Use plugin registration for new suite-level workload/scheduler keys.
- Keep plugin modules tiny (`register(registry)` only).
- `cli_token` must map to real simulator flags accepted by `apps/main.cpp`.

## What not to do in v1

- Do not add strict metric-regression CI gates yet.
- Do not add heavyweight governance/versioning layers unless needed by active users.
- Do not make report generation a hard dependency for basic benchmark execution.
